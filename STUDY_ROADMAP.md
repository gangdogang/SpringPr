# Spring 게시판 백엔드 학습 로드맵

## 1. 현재 위치

현재 프로젝트에서 경험한 내용은 다음과 같다.

- Spring MVC의 Controller–Service–Repository 계층 분리
- Spring Data JPA를 이용한 게시글 CRUD
- Entity 연관관계와 지연 로딩
- 요청·응답 DTO 분리와 Bean Validation
- `@Transactional`과 JPA 변경 감지
- 전역 예외 처리
- fetch join을 이용한 N+1 문제 대응
- 조회수 동시성 문제와 원자적 UPDATE 학습
- 댓글과 좋아요 도메인의 기본 구조 설계

현재 단계는 **기본 CRUD를 구현한 뒤, 안정적인 백엔드로 확장하는 단계**다. 앞으로는 기능 개수를 빠르게 늘리기보다, 하나의 기능을 설계·구현·테스트·복습까지 끝내는 방식으로 공부한다.

## 2. 학습 원칙

### 직접 구현하기

- 먼저 스스로 요구사항과 흐름을 작성한다.
- 막혔을 때는 전체 정답보다 개념 설명이나 부분 힌트를 먼저 확인한다.
- 코드를 붙여 넣기 전에 각 애너테이션과 쿼리가 필요한 이유를 설명할 수 있어야 한다.

### 작은 단위로 완성하기

기능 하나를 다음 순서로 진행한다.

1. 요구사항 작성
2. API와 예외 상황 설계
3. 정상 동작 구현
4. 실패 상황 테스트
5. 리팩터링
6. 배운 내용 기록

### 완료 기준을 테스트로 확인하기

- 정상 요청만 확인하고 끝내지 않는다.
- 존재하지 않는 리소스, 잘못된 입력, 중복 요청을 함께 확인한다.
- 테스트가 실패하면 원인을 설명한 후 수정한다.

### 학습 기록 남기기

각 기능이 끝날 때 아래 내용을 5줄 이내로 기록한다.

- 어떤 문제가 있었는가?
- 처음에는 어떻게 구현했는가?
- 어떤 문제가 발생했는가?
- 최종적으로 어떤 방법을 선택했는가?
- 다른 방법과 비교했을 때 장단점은 무엇인가?

## 3. 전체 학습 순서

## 1단계: 댓글 CRUD와 연관관계

### 학습 목표

- 하나의 기능을 Controller부터 Repository까지 완성한다.
- 다대일 연관관계와 연관된 데이터의 생명주기를 이해한다.
- REST API의 URL과 HTTP 상태 코드를 설계한다.

### 구현 순서

- [ ] 댓글 생성
- [ ] 게시글별 댓글 목록 조회
- [ ] 댓글 수정
- [ ] 댓글 삭제
- [ ] 댓글 요청·응답 DTO 분리
- [ ] 입력값 검증과 예외 처리
- [ ] 게시글 삭제 시 댓글 처리 정책 결정

### 권장 API

| 기능 | Method | URL | 성공 상태 |
|---|---|---|---|
| 댓글 생성 | POST | `/api/posts/{postId}/comments` | 201 |
| 댓글 목록 | GET | `/api/posts/{postId}/comments` | 200 |
| 댓글 수정 | PUT 또는 PATCH | `/api/comments/{commentId}` | 204 |
| 댓글 삭제 | DELETE | `/api/comments/{commentId}` | 204 |

### 공부할 개념

- 연관관계의 주인
- `@ManyToOne(fetch = LAZY)`
- 영속성 컨텍스트와 변경 감지
- cascade와 orphan removal
- DTO에서 Entity를 직접 노출하면 안 되는 이유
- `POST`, `PUT`, `PATCH`, `DELETE`의 의미

### 완료 기준

- [ ] 없는 게시글에 댓글을 작성하면 404가 반환된다.
- [ ] 없는 사용자가 작성하면 404가 반환된다.
- [ ] 빈 댓글은 400이 반환된다.
- [ ] 댓글 목록에서 작성자 닉네임을 확인할 수 있다.
- [ ] 댓글 수정과 삭제 시 존재 여부를 확인한다.
- [ ] 게시글을 삭제할 때 댓글을 어떻게 처리할지 설명할 수 있다.

### 복습 질문

1. `Comment`가 `Post`와 `User`를 객체로 참조하는 이유는 무엇인가?
2. 댓글 조회 중 N+1 문제가 생길 수 있는 지점은 어디인가?
3. 게시글 삭제 시 cascade를 사용할 때의 장단점은 무엇인가?

### 학습 목표

- 직접 HTTP 요청만 보내는 방식에서 자동화 테스트로 이동한다.
- 단위 테스트, 계층 테스트, 통합 테스트의 차이를 구분한다.

### 구현 과제

- [ ] 게시글 Service 테스트
- [ ] 댓글 Service 테스트
- [ ] Controller Validation 테스트
- [ ] Repository 쿼리 테스트
- [ ] 예외 응답 테스트
- [ ] 각 테스트에서 필요한 데이터를 직접 생성

### 공부할 개념

- JUnit 5와 AssertJ
- Given–When–Then 패턴
- `@SpringBootTest`
- `@WebMvcTest`와 MockMvc
- `@DataJpaTest`
- Mockito와 mock의 사용 범위
- 테스트 격리와 fixture

### 테스트 우선순위

1. 서비스의 정상 흐름
2. 존재하지 않는 데이터
3. Validation 실패
4. 중복 요청

- [ ] 테스트 하나가 다른 테스트의 실행 결과에 영향을 주지 않는다.
- [ ] 실패 메시지만 보고 어떤 요구사항이 깨졌는지 알 수 있다.
- [ ] `./gradlew test`로 모든 테스트를 한 번에 확인할 수 있다.

## 3단계: 좋아요와 중복 요청

### 학습 목표

- DB 제약조건과 애플리케이션 검증의 차이를 이해한다.
- 동일한 요청이 반복되거나 동시에 들어올 때의 동작을 설계한다.

### 구현 과제

- [ ] 좋아요 추가
- [ ] 좋아요 취소
- [ ] 사용자별 좋아요 여부 조회
- [ ] 동일 사용자의 중복 좋아요 방지
- [ ] 좋아요 수 조회
- [ ] 동시에 좋아요를 요청하는 테스트

### 공부할 개념

- 유니크 제약조건
- `DataIntegrityViolationException`
- 멱등성
- DB의 `count` 결과를 사용할지 별도 카운터를 저장할지 비교
- 경쟁 조건과 트랜잭션 경계

### 설계 질문

1. 좋아요 추가 API를 두 번 호출하면 어떤 결과가 되어야 하는가?
2. 애플리케이션에서 중복 여부를 먼저 조회해도 DB 유니크 제약이 필요한 이유는 무엇인가?
3. `Post.likeCount`와 실제 좋아요 행의 개수가 달라질 가능성은 없는가?

### 완료 기준

- [ ] 한 사용자는 같은 게시글에 한 번만 좋아요를 누를 수 있다.
- [ ] 좋아요 취소를 반복해도 데이터가 잘못되지 않는다.
- [ ] 중복 요청의 HTTP 응답 정책을 설명할 수 있다.

## 4단계: 트랜잭션과 동시성 심화

### 학습 목표

- `@Transactional`이 모든 동시성 문제를 해결하지 않는 이유를 이해한다.
- 문제 유형에 따라 원자적 UPDATE, 낙관적 락, 비관적 락을 선택한다.

### 실습 순서

- [ ] 기존 조회수 갱신 유실을 다시 재현
- [ ] 원자적 UPDATE 쿼리로 해결
- [ ] 별도 실습에서 `@Version`을 이용한 낙관적 락 적용
- [ ] 비관적 락 적용 후 대기 시간 확인
- [ ] 세 방식의 쿼리 수와 동작 비교

### 선택 기준

| 방식 | 적합한 상황 | 주의점 |
|---|---|---|
| 원자적 UPDATE | 조회수, 단순 카운터 증감 | 영속성 컨텍스트와 DB 값 불일치 가능성 |
| 낙관적 락 | 충돌이 드문 수정 | 충돌 시 예외 처리와 재시도 필요 |
| 비관적 락 | 충돌이 많고 순차 처리가 필요한 작업 | 대기 시간, 처리량 저하, 데드락 가능성 |

### 공부할 개념

- 갱신 유실
- 트랜잭션 격리 수준
- DB row lock
- `@Version`
- `PESSIMISTIC_WRITE`
- JPQL 벌크 연산과 영속성 컨텍스트
- 재시도 정책

### 완료 기준

- [ ] 조회수 동시 증가 테스트가 반복 실행되어도 통과한다.
- [ ] 세 방식 중 조회수에 UPDATE 쿼리를 선택한 이유를 설명할 수 있다.
- [ ] 동시성 테스트 내부의 작업 예외를 놓치지 않는다.

## 5단계: 목록 API와 JPA 성능

### 학습 목표

- 데이터가 많아져도 사용할 수 있는 목록 API를 만든다.
- 실행되는 SQL을 보고 조회 성능 문제를 판단한다.

### 구현 과제

- [ ] 게시글 페이징
- [ ] 최신순·조회수순·좋아요순 정렬
- [ ] 제목 검색
- [ ] 댓글 목록 페이징
- [ ] 목록용 DTO와 상세용 DTO 분리
- [ ] 조회 쿼리 수 확인

### 공부할 개념

- `Page`, `Pageable`, `Slice`
- offset pagination의 특징
- fetch join과 pagination의 주의점
- N+1 문제
- DTO projection
- DB index 기초

### 완료 기준

- [ ] 게시글 전체를 한 번에 메모리로 가져오지 않는다.
- [ ] 목록 조회에서는 불필요한 본문 전체를 반환하지 않는다.
- [ ] 페이지 크기에 따라 쿼리 개수가 증가하지 않는다.

## 6단계: 인증과 인가

### 학습 목표

- 요청으로 전달받은 사용자 ID를 신뢰하지 않는다.
- 로그인 사용자와 리소스 작성자를 기준으로 권한을 검사한다.

### 구현 순서

- [ ] 회원가입
- [ ] 비밀번호 암호화
- [ ] 로그인
- [ ] 인증 사용자 조회
- [ ] 게시글·댓글 작성자 자동 지정
- [ ] 작성자만 수정·삭제 가능하도록 인가

### 공부할 개념

- Spring Security filter chain
- 인증과 인가의 차이
- `PasswordEncoder`
- 세션 방식과 JWT 방식 비교
- SecurityContext
- 401과 403의 차이

### 완료 기준

- [ ] 게시글 생성 요청에 `writerId`를 받지 않는다.
- [ ] 로그인하지 않은 사용자는 보호된 API를 호출할 수 없다.
- [ ] 다른 사용자의 게시글과 댓글을 수정·삭제할 수 없다.
- [ ] 비밀번호를 평문으로 저장하지 않는다.

## 5. 한 기능을 공부하는 일일 루틴

### 시작 전 10분

- 오늘 구현할 요구사항을 한 문장으로 작성한다.
- 정상 상황과 실패 상황을 각각 적는다.
- 어떤 계층이 변경될지 예상한다.

### 구현

- 가장 작은 정상 흐름부터 구현한다.
- 기능이 동작하면 실패 상황을 추가한다.
- 실행되는 SQL과 HTTP 응답을 확인한다.

### 테스트

- 정상 케이스를 자동화한다.
- 없는 데이터와 잘못된 입력을 테스트한다.
- 이전 기능이 깨지지 않았는지 전체 테스트를 실행한다.

### 마무리 10분

- 오늘 새로 알게 된 개념을 3개 적는다.
- 이해하지 못한 부분을 질문 형태로 남긴다.
- 다음 작업을 체크박스 하나로 정한다.

## 6. 코드 리뷰 체크리스트

### Controller

- [ ] Entity를 요청이나 응답에 직접 사용하지 않는가?

### Service

- [ ] 트랜잭션 범위가 적절한가?
- [ ] 비즈니스 규칙이 Controller에 흩어져 있지 않은가?

- [ ] 동시 요청이 들어왔을 때 문제가 없는가?

- [ ] 불필요한 쿼리가 실행되지 않는가?

- [ ] 필요한 DB 제약조건이 있는가?
- [ ] 많은 데이터를 한 번에 조회하지 않는가?

### Test

- [ ] 테스트 이름이 요구사항을 설명하는가?
- [ ] 테스트끼리 독립적인가?
- [ ] 정상·실패·경계 조건을 확인하는가?

## 9. 최종 목표

이 프로젝트를 마쳤을 때 다음 질문에 코드와 테스트로 답할 수 있어야 한다.

1. 요청 하나가 Controller부터 DB까지 어떻게 처리되는가?
2. 트랜잭션은 어디에서 시작하고 끝나는가?
3. JPA는 언제 SQL을 실행하는가?
4. 동시에 요청이 들어오면 어떤 데이터 문제가 발생할 수 있는가?
5. 사용자의 입력과 권한을 어디에서 검증하는가?
6. 데이터가 많아졌을 때 조회 성능을 어떻게 개선하는가?
7. 변경한 기능이 기존 기능을 깨뜨리지 않았다는 것을 어떻게 증명하는가?
8. 새로운 환경에서 애플리케이션과 DB를 어떻게 재현하는가?

이 질문에 설명할 수 있으면 단순 CRUD 구현을 넘어, 실제 백엔드 개발의 기본기를 갖춘 상태라고 볼 수 있다.
